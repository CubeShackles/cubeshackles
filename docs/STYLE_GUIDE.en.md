# Style Guide — English

[English](STYLE_GUIDE.en.md) | [Português](STYLE_GUIDE.pt.md)

**Owner: CubeShackles (founder-led).**

CubeShackles English documentation already has a strong, consistent voice
across the umbrella repo (`README.md`, `PRODUCTION_PRINCIPLES.md`,
`SYSTEM_ARCHITECTURE.md`). This guide names what that voice already does, so
new and translated documentation matches it.

1. **Precise verbs for maturity.** Use "implements" / "provides" / "active"
   only for what exists and is exercised by tests. Use "designed for" /
   "intended to" / "scaffolded" for committed direction with structure on
   disk. Use "planned" only when the repository does not yet exist. See
   `REPOSITORY_MAP.md`'s status vocabulary and
   [`LOCALIZATION_POLICY.md`](LOCALIZATION_POLICY.md) §4.
2. **State what a thing is not.** Every README says what the repository is
   NOT, immediately after saying what it is (see `documentation-templates.md`).
3. **No fake decentralization or unearned readiness language.** No
   claims of production-readiness, full operationality, uptime guarantees,
   or unbreakability outside the two allow-listed doctrine documents.
4. **Institutional audience, not marketing audience.** Write for an engineer,
   regulator, or institutional reviewer deciding whether to trust a claim —
   not for a landing-page visitor. Short declarative sentences over
   adjectives.
5. **Boundary language.** State explicitly what a repository "owns" and
   "does not own" (see `cubeshackles-compliance-engine/README.md` for the
   pattern) — this is the same discipline as bilingual claims discipline in
   [`DOCUMENTATION_STANDARD.md`](DOCUMENTATION_STANDARD.md) §3.
6. **No AI-vendor authorship framing.** Per
   [`../governance/policies/authorship-and-tooling.md`](../governance/policies/authorship-and-tooling.md).
